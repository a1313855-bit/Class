# 完整網站平台開發 SOP 檢查清單
> 客製化技術棧:前端(HTML/CSS/JS/jQuery/Bootstrap/Vue/React)、後端(Java/Python)、資料庫(MySQL/PostgreSQL)、數據分析、n8n、K8s

---

## 0. 技術棧選型建議(先決定,再開工)

你列的技術範圍很廣,實際專案不建議「全部都用」,先做以下決策:

| 決策項目 | 建議 |
|---|---|
| 前端框架 | 新專案用 **Vue 3** 或 **React**,jQuery+Bootstrap 適合舊系統維護或極簡管理後台,不建議新專案混用 |
| 後端語言 | Java (Spring Boot) 適合**大型、多人協作、強型別、企業級**系統;Python (Django/FastAPI) 適合**開發速度快、需結合數據分析/AI**的專案 |
| 資料庫 | MySQL 適合**電商/一般CRUD**;PostgreSQL 適合**複雜查詢、JSON欄位、地理資訊、數據分析用途** |
| 數據分析 | 若後端選 Python,數據分析(pandas/numpy)可與後端共用同一語言,開發效率最高 |
| n8n | 用於**跨系統自動化**(例如:表單送出 → 寫入DB → 發Line通知 → 寫入Google Sheet),不要拿來取代核心後端邏輯 |
| K8s | 只有當你有**多服務、需要自動擴縮容、多環境部署**時才需要;單一小型網站用 Docker Compose 即可,K8s 會增加維運成本 |

**建議一個具體組合範例(可依需求調整):**
```
前端:Vue 3 + Vite + Bootstrap(快速)或 Tailwind(客製化強)
後端:Java Spring Boot(核心交易系統) + Python FastAPI(數據分析/AI微服務)
資料庫:PostgreSQL(主資料庫)+ Redis(快取)
自動化:n8n(串接第三方通知、報表、爬蟲排程)
部署:Docker → K8s(正式環境)
```

## 3. 架構設計規則(MVC + DAO)

### 3.1 後端分層(Java 範例)
```
controller/   → 接收 HTTP request,呼叫 service,回傳 DTO
service/      → 商業邏輯層(交易規則、驗證邏輯)
dao/(或 repository/) → 純資料存取(Spring Data JPA / MyBatis)
model/(entity/) → 資料表對應的物件
dto/          → 前後端資料傳輸格式(不要直接把 entity 回傳給前端)
config/       → 資料庫連線、安全性、跨域設定
```

### 3.2 後端分層(Python/FastAPI 範例)
```
routers/      → 相當於 controller
services/     → 商業邏輯
repositories/ → 相當於 DAO,負責 SQL/ORM 操作
models/       → SQLAlchemy ORM 定義
schemas/      → Pydantic 驗證用的資料格式(相當於 DTO)
```

### 3.3 規則重點
- [ ] Controller 不寫 SQL、不寫商業邏輯
- [ ] DAO 只做資料存取,不判斷商業規則
- [ ] 統一錯誤處理(Java: `@ControllerAdvice`;Python: exception handler)
- [ ] API 回傳格式統一(建議固定格式:`{ code, message, data }`)
- [ ] 前後端分離,API 走 RESTful 或視需求走 GraphQL

## 4. 需求功能規劃

### 4.1 基礎功能清單
- [ ] 會員系統(註冊/登入/JWT驗證/權限分級 RBAC)
- [ ] CRUD 模組(含分頁、搜尋、篩選、排序)
- [ ] 檔案上傳(圖片/文件,建議搭配物件儲存如 S3 / MinIO)
- [ ] 通知系統(Email、n8n 串接 Line/Slack 通知)
- [ ] 後台管理介面(資料管理、報表)

### 4.2 數據分析模組(若有需求)
- [ ] 用 Python(pandas/numpy/matplotlib 或 Plotly)建立獨立分析服務
- [ ] 分析結果可存回 PostgreSQL 或透過 API 提供前端視覺化(可用 ECharts/Chart.js)
- [ ] 排程任務(定期跑報表)可用 n8n 排程觸發,或用 Celery(Python)/ Quartz(Java)

### 4.3 n8n 自動化情境範例
- [ ] 表單提交 → n8n webhook → 寫入資料庫 → 發送通知
- [ ] 每日排程 → 呼叫後端 API 拉資料 → 產生報表 → Email 寄送
- [ ] 第三方系統(如金流、簡訊)串接,降低後端開發負擔

### 4.4 規格文件
- [ ] 畫 ER 圖(資料表關聯)
- [ ] API 規格書:Java 用 Swagger(springdoc-openapi);Python 用 FastAPI 內建 Swagger UI

## 7. 安全性檢查(容易被忽略但必做)

- [ ] SQL Injection 防護(使用 ORM/參數化查詢,不要字串拼接 SQL)
- [ ] XSS/CSRF 防護(前端輸出過濾、後端 CORS 設定)
- [ ] 密碼加密:bcrypt / argon2,不可明文儲存
- [ ] JWT 或 Session 過期機制、Refresh Token 機制
- [ ] API Rate Limiting(可用 n8n 或後端中介層做)
- [ ] HTTPS + SSL 憑證(Let's Encrypt 或雲端服務商憑證)
- [ ] 資料庫帳號權限最小化(不要用 root 帳號跑應用程式)

---

## 8. 容器化與部署(Docker → K8s)

### 8.1 Docker 化
- [ ] 每個服務(前端、Java後端、Python後端、n8n)獨立 Dockerfile
- [ ] 用 multi-stage build 縮小 image 體積
- [ ] 本地先用 docker-compose 驗證整合是否正常

### 8.2 K8s 部署重點
- [ ] 撰寫 Deployment / Service / Ingress YAML
- [ ] ConfigMap / Secret 管理環境變數與敏感資訊
- [ ] HPA(Horizontal Pod Autoscaler)設定自動擴縮容規則
- [ ] 使用 Helm Chart 管理多環境部署設定(dev/staging/prod)
- [ ] 資料庫建議不放進 K8s 內管理(用雲端 RDS 服務較穩定),除非有專門 DBA 維運能力

### 8.3 CI/CD Pipeline
- [ ] GitHub Actions / GitLab CI:push → build image → push 到 Registry → K8s 自動部署(可用 ArgoCD 做 GitOps)
- [ ] 建立 staging 環境先驗證,再手動或自動 promote 到正式環境
